## W001 · 设计文档权威制

| 项 | 内容 |
|---|------|
| **规则** | 所有经老板确认的设计沟通结果，必须同步写入 `docs/decisions.md` 和 `docs/design.md` |
| **PM 文档同步** | PM 必须在每次讨论/会议结束后，将结论同步至对应文档：① 项目级决策 → `decisions.md` ② 角色职责/规格/设计 → `design.md` ③ 优化需求 → `optimizations.md` ④ 里程碑/进度变更 → `progress.md`。**漏同步 = 决策未生效，任何人有权拒绝执行** |
| **开发纪律** | 开发模式下，代码实现严格遵循 MD 文档中已确认的设计内容。**禁止新增、修改、替换** MD 中已有的设计思路 |
| **变更流程** | 如需变更设计 → 先开会讨论 → 老板确认 → 更新 MD 文档 → 再写代码 |
| **文件入库** | 任何新增物理文件（.bat / .py / .json / 配置 / 资源）必须在 `changelog.md` 记录新建条目。项目根目录下的启动/工具脚本须同时记入 `progress.md`「根目录文件清单」表。**漏录 = 决策未生效** |

## W002 · 交互逻辑参照 Undertale

| 项 | 内容 |
|---|------|
| **规则** | 这是一款面向普通人类玩家的纯文字冒险游戏，交互逻辑参照 Undertale 的设计哲学 |
| **具体标准** | ① 界面简洁直观，不堆砌元素 ② 操作符合人类习惯（居中菜单、空格/回车翻页、鼠标悬停反馈） ③ 不过度依赖键盘快捷键 ④ 加适当操作提示（如"▼ 空格/回车继续"） |
| **UI 验收前置** | 每个功能模块完成后，UI 必须先以人类玩家视角验证——所有交互元素在默认窗口尺寸下可见可达、操作提示清晰、悬停反馈正常。UI 验证通过后再移交 QA |
| **自检问题** | 每次交互改动前问：*Undertale 会怎么做？* |
| **W002-4 · 面板文本不溢出** | 面板内所有 `Label`/`Text` 控件文本在面板宽度内必须完全可见。`wraplength` 在控件创建时即设置，不得依赖默认行为。面板 reopen 时宽度应重新计算。UI 自检项：面板打开 → 每个物品悬停 → 确认描述文字完整可读不截断。 |
| **W002-5 · 打字/快进双路径渲染一致** | `_typewrite_char`（打字路径）和 `_skip_typing`（快进重建路径）对相同标记文本必须产生相同的视觉输出——同色、同行距、同顺序。任何标记（`[key]`、`[shake]` 等）的新增必须在两条路径同时验证。UI 自检项：逐字播完一段含标记文本 → 下一段快进跳过 → 确认颜色/行距/换行完全一致无跳变。依据 B032（key 文本快进时消失且乱序）、B035（翻页行距不一致）。 |

## W003 · QA 验收前置

| 项 | 内容 |
|---|------|
| **规则** | 每个功能模块完成后，QA 必须先内部验证通过，再提交给老板评审 |
| **QA 全程介入** | QA 在每个功能验收环节都必须介入验收 |
| **Bug 清单** | 老板发现 Bug → QA 记录（编号+复现步骤+优先级）→ FD 修复 → QA 出验证记录 → 老板物理验收关闭。FD 不得自行将 Bug 标记为已修复。维护者见 `docs/bug_list.md` |
| **QA 两层分工** | **逻辑层**（AI）：跑全量自动化测试 + 模拟点击状态校验 + FD 自检声明二次确认 + Bug 复现记录。**物理层**（老板）：肉眼验证视觉效果/动画/音效/排版，确认修复后关闭。QA 不放行——QA 提供验证记录给老板做决策依据 |
| **优化督促** | QA 负责督促 FD 完成 `docs/optimizations.md` 中全部"重要优化" |
| **优化打勾** | 重要优化由 QA 完成验收后打勾；PM 仅提优化意见，表格记录维护移交 QA |
| **QA 检查项** | 流程可达性、变量正确性、边界情况、死链检查、缺失提示（"未完待工"）、文本错字 |
| **放行标准** | 只有 QA 确认全部检查项通过后，才移交给老板 |
| **FD 交底** | FD 交付功能时必须附带一句功能行为说明（如"GM 加载章节：已加载的章节再点击不会重复追加"），QA 据此验证并写入测试矩阵。无交底说明 = QA 有权拒绝验收 |
| **FD 自检声明** | FD 修复 Bug 后，必须附带一句自检声明（格式："已物理验证：双击 bat、按 Tab 打开面板、按空格翻页、点存档标签，无异常"）。无自检声明 = QA 拒收。此条依据 D033 会议决议，是对 `agents.md` FD P4 的强制执行 |
| **ND 变更通知** | ND 每完成或修改一章 JSON，须附带简述交付 QA（格式："ch04: 新增 12 节点，[shake] 2 处、[pause] 3 处、multi_click 1 处、MG4 触发点 1 处"）。QA 据此同步跑测计划。无通知 = QA 有权拒绝新增该章验收 |
| **ND 自检声明** | ND 交付章节 JSON 时须附带自检声明（格式见 `docs/nd_selfcheck.md`）。无自检声明 = QA 拒收该章验收 |

## W006 · 异步回调防卫（B019 MG2 竞态复盘）

| 项 | 内容 |
|---|------|
| **规则** | Tkinter `after()` 回调必须防守重入。所有涉及计时器的代码遵循：**cancel 旧 → 设状态 → schedule 新**。同一函数被多条 `after` 路径调用时，函数内部必须先清掉可能已排队的其他调用。 |
| **三项标准** | ① 每个 `after()` 调度前，必须 cancel 同源旧计时器 ID ② 超时回调执行后，立即将关联的计时器 ID 置为 None（标记"已触发"，防止后续误 cancel）③ 多条路径汇聚到同一回调函数时，函数内部第一行 cancel 所有可能排队的旧计时器 |
| **自检** | FD 每次使用 `after()` 的代码改动，自检清单追加一条：验证快速连续操作（如光点消失边缘点击）不会导致回调重复执行 |
| **依据** | B019 — MG2 光点叠层：`_hit_dot` 与 `_miss_dot` 各调度 `after(200ms, _clear_dot_and_next)`，竞态下 `_spawn_flash` 被重复调用 |

## W007 · 小游戏生命周期防卫（MG4A 崩溃复盘）

| 项 | 内容 |
|---|------|
| **规则** | 小游戏完成回调（`_complete → callback → destroy`）不得在渲染循环的调用栈内同步执行。`_complete` 必须将回调延迟到循环安全返回之后（`after`/`after_idle`）。`destroy` 必须无条件执行清理，不得依赖 `_running` 等中间状态决定是否调用 `_on_stop`。 |
| **三条标准** | ① `_complete` 内的 `callback` 通过 `after(10, ...)` 延迟，确保 `_loop`/`_draw_frame` 安全返回后再触发销毁 ② `destroy` 无条件调用 `_on_stop`（取消 timers、unbind events）后再 `canvas.destroy()` ③ `_on_stop` 始终执行完整的 timer/event 清理，不得有条件 return |
| **自检** | FD 每次修改小游戏代码后，自检清单追加一条：MG4A 海鸟躲避连中 3 次 → 确认无 `NoneType` 崩溃，canvas 安全销毁 |
| **依据** | MG4A 崩溃链：`_check_hits → _complete → callback → destroy(canvas=None)` 在 `_loop` 调用栈内同步执行，后续 `self.canvas.after(...)` 访问 None |


## W008 · Tkinter 布局防卫（MG4B 全黑 + 气泡不显示 复盘）

| 项 | 内容 |
|---|------|
| **规则 ①** | `winfo_width()` / `winfo_height()` 在 widget 首次 `place()` 后、Tkinter 主循环渲染前返回 `1`（非 falsy），不可用 `or DEFAULT` 进行尺寸兜底。所有 Canvas/widget 的尺寸获取必须使用 `if w < 50: w = DEFAULT` 阈值判断。 |
| **规则 ②** | `place(in_=pack_widget)` 跨几何管理器定位不可靠——`pack()` 管理的 widget 没有稳定的像素坐标暴露给 `place()`。所有需相对定位的 widget 应置于同一父容器内，使用同一种几何管理器（统一 pack 或统一 place）。 |
| **自检** | FD 每次新建 Canvas 或覆盖层 widget 时：① 获取尺寸用的是 `< 50` 阈值而非 `or default`；② 定位逻辑不使用 `place(in_=pack_widget)` 跨管理器引用 |
| **依据** | MG4B 全黑 — `winfo_width()=1` → Canvas 1×1 px。气泡不显示 — `place(in_=self._tab_hint)` 在 pack 布局下坐标不可靠，气泡从未出现在可视区域 |


## W009 · 分支对称性（结局循环复盘）

| 项 | 内容 |
|---|------|
| **规则** | 同一状态判断逻辑存在于多个调用路径时，必须确认各路径实现一致。新增路径时须审计已有路径的同类逻辑，不得遗漏。 |
| **自检** | FD 每次新增 `cmd_*` / `_on_*` 回调，检查是否有其他路径在同类状态下执行了额外逻辑 |
| **依据** | ch06 结局循环 — `is_ending_node` 转发仅在 `_on_text_done` 实现，`cmd_choice` 遗漏。同类：B018 / MG4A `_on_stop` / `_complete` 同步回调——均为一逻辑两路径不对称 |


## W010 · 状态隔离防卫（B025~B028 复盘）

| 项 | 内容 |
|---|------|
| **规则** | ① JSON 数据文件写入后必须立即验证合法性（`json.load`），不得以 `try/except` 静默吞错。② 对象状态（缓存、标志位）必须在 `__init__` 中初始化，不得依赖 UI widget 的生命周期——widget 会被 `destroy()` 重建。③ 任何 widget 操作前必须 `winfo_exists()` 守卫。④ `after_cancel` 只 cancel 已有的有效 ID（判真再 cancel），不得用 `getattr(x, 0)` 等默认值。 |
| **自检** | FD 每次新增 JSON 数据文件：写入后立即 `json.load` 验证。FD 每次新增 widget 相关属性：确认是否应在 `__init__` 初始化。FD 每次访问可能在面板关闭时被销毁的 widget：加 `winfo_exists()` + `try/except TclError`。FD 每次 `after_cancel`：确认变量可能为 None/0，先判真。 |
| **依据** | B025 — diary.json 内层引号破坏 JSON → 静默失败日记不显示。B026 — `_diary_cache` 绑定到面板 widget 生命周期 → `cmd_gm_load_chapter` 面板未打开时 crash。B027 — 面板关闭后 `_render_diary` 访问已销毁 widget → TclError。B028 — `after_cancel(getattr(0))` 传 0 → ValueError。四 bug 根因相同：数据/状态/控件三层生命周期未隔离。 |


## W011 · 物品效果穿透

| 项 | 内容 |
|---|------|
| **规则** | 物品 `carry_effects` 必须穿透到游戏逻辑层——结局判定(`check_ending`)、GM 面板变量显示、选项条件判定——不得仅停留在面板展示文字。任何修改变量值的物品效果必须在所有读取该变量值的代码路径中可见。 |
| **自检** | FD 每次新增物品携带效果后：① 确认 `check_ending()` 的有效值计算包含了该效果；② 确认 GM 面板（Ctrl+Shift+G）变量显示带物品修饰（绿色 +N / 红色 -N）；③ 确认该效果能真实改变结局走向（GM 预设测试）。 |
| **依据** | M7 物品系统上线后，`key` 的 survival_will+2 和 `evil_wood_carving` 的 sanity-2 仅显示在面板文字中，`check_ending()` 读 StateManager 原始值未包含物品效果，导致物品无法影响结局判定。修复：`check_ending()` 计算有效值 = 原始值 + `item_mgr.get_carry_effects()`。 |

## W005 · 角色交付不可绕过 + 声明确认制

| 项 | 内容 |
|---|------|
| **规则** | 每个角色完成交付时，必须在输出中显式输出规定格式的**自检/验收声明**。PM 只能在收到此声明后方可在 `progress.md` / `todowrite` 中将对应任务标记为已完成。**无声明的完成 = 工作从未发生**。 |
| **PM 放行前验声明** | PM 每次将 `todowrite` 某任务标为 `completed` 前，必须先在上下文中确认已收到该角色的规定格式声明。无声明即放行 = PM 越权。 |
| **声明格式表** | 见下方角色→声明格式映射表。格式是强制性的，不可简写或省略关键字段。 |
| **角色隔离挡板** | PM 不可自行写业务逻辑代码。如需以非 PM 角色执行任务，必须在输出中先声明 `[角色名] 开始工作`，然后输出该角色的交付格式声明。 |

### 各角色交付声明格式（强制）

| 角色 | 必须输出的声明 |
|------|--------------|
| **FD** | `FD自检：已物理验证 — 双击bat启动、{具体操作序列}、{观察结果}，无异常。{若有crash: 发现并修复crash n个：{简述}}→ 移交QA` |
| **ND** | `ND自检：{章}章修改，新增/修改 {n} 节点，[shake]{x}处/[pause]{y}处/[sound]{z}处。汇聚节点{c}个，text_bridges {b}条全配置，*fallback 到位。已按 nd_selfcheck.md 自检通过（含 bridges.md 对表）。→ 移交QA` |
| **SD** | `SD自检：音效 {name} 修改，wav 文件已确认存在，play()/stop() 行为正确，不阻塞主线程。→ 移交QA` |
| **QA** | `QA验收：{验收项}。测试输出：{run_all.py 实际结果，如 87/87 PASS, 0 FAIL}。FD声明一致性：{FD声称验证了X，对应测试Y 已跑/未跑}。数据完整性：{validate_data 结果，节点数/引用闭合情况}。物理操作记录：{鼠标点击/键盘输入序列}。结论：{通过/不通过}。→ 移交PM` |
| **PM** | 不可自行写代码。放行任务前必须输出：`PM放行：已收到{角色}的验收声明，{任务}通过，更新进度。` |
